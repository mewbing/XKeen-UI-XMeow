**vite.config.ts -- add separate proxy for Flask Config API:**

Currently, ALL `/api/*` requests route to mihomo at `http://172.16.10.1:9090`. Flask backend runs on port 5000 and handles `/api/service/*`, `/api/versions`, `/api/config`, `/api/xkeen/*`, `/api/health`.

Replace the single proxy rule with two rules. Order matters -- more specific routes MUST come first:

```typescript
server: {
  proxy: {
    // Flask Config API (service management, config, xkeen files, versions, health)
    '/api/service': {
      target: 'http://172.16.10.1:5000',
      changeOrigin: true,
    },
    '/api/versions': {
      target: 'http://172.16.10.1:5000',
      changeOrigin: true,
    },
    '/api/config': {
      target: 'http://172.16.10.1:5000',
      changeOrigin: true,
    },
    '/api/xkeen': {
      target: 'http://172.16.10.1:5000',
      changeOrigin: true,
    },
    '/api/health': {
      target: 'http://172.16.10.1:5000',
      changeOrigin: true,
    },
    // Mihomo REST API (all other /api/* goes to mihomo)
    '/api': {
      target: 'http://172.16.10.1:9090',
      changeOrigin: true,
      headers: {
        'Authorization': 'Bearer admin',
      },
    },
  },
},
```

This way, `/api/service/start`, `/api/config`, etc. go to Flask on port 5000, while `/api/version` (mihomo), `/api/traffic`, `/api/connections`, etc. go to mihomo on port 9090.

Note: Flask backend does NOT need the Authorization bearer header (it has no auth). Only mihomo needs it. This is why Flask routes must NOT inherit the Authorization header from the catch-all `/api` rule.

**config-api.ts -- add res.ok check to fetchServiceStatus (line 38-41):**

Currently `fetchServiceStatus()` calls `res.json()` without checking `res.ok`. If the server returns a non-200 response (or if the request hits mihomo instead of Flask), parsing may fail silently.

Change to:
```typescript
export async function fetchServiceStatus(): Promise<{
  running: boolean
  pid: number | null
}> {
  const res = await fetch(`${getBaseUrl()}/api/service/status`, {
    signal: AbortSignal.timeout(5000),
  })
  if (!res.ok) {
    throw new Error(`Service status request failed: ${res.status}`)
  }
  return res.json()
}
```

Also add the same `res.ok` check to `fetchVersions()` (line 51-54):
```typescript
export async function fetchVersions(): Promise<{
  xkeen: string
  dashboard: string
}> {
  const res = await fetch(`${getBaseUrl()}/api/versions`, {
    signal: AbortSignal.timeout(5000),
  })
  if (!res.ok) {
    throw new Error(`Versions request failed: ${res.status}`)
  }
  return res.json()
}
```

These throw on non-ok responses, which are then caught by the existing try/catch in useServiceStatus hook and OverviewPage. This prevents silent failures and makes debugging easier.
