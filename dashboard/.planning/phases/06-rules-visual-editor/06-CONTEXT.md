# Phase 6: Rules Visual Editor - Context

**Gathered:** 2026-03-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Визуальное редактирование секции `rules:` из config.yaml с drag-and-drop приоритизацией. Правила отображаются как карточки-блоки, которые можно перемещать, раскрывать, редактировать. Изменения сохраняются обратно в config.yaml.

Не входит в фазу: редактирование proxy-groups (фаза 7), rule-providers.

</domain>

<decisions>
## Implementation Decisions

### Группировка правил в блоки
- Реализовать **3 варианта группировки** с переключателем:
  1. По proxy-group — каждый блок = одна целевая группа (YouTube, Telegram, DIRECT...)
  2. По секциям конфига — блоки повторяют логические секции YAML (комментарии-разделители: «Исключения», «ADULT», «Сервисы», «RU», «Cloud-сети»)
  3. Двухуровневая — секции как категории, внутри карточки по proxy-group
- Пользователь выберет предпочтительный вариант после визуальной оценки

### Плотность информации на карточке
- Реализовать **2 режима** с переключателем:
  1. Минимально — имя блока, кол-во правил, целевая proxy-group
  2. Подробно — имя, кол-во правил, proxy-group + превью первых 2-3 правил (DOMAIN-SUFFIX, GEOSITE...)
- Пользователь выберет после оценки

### Раскрытие блока
- Таблица правил внутри раскрытого блока
- Колонки: тип (DOMAIN-SUFFIX, GEOSITE...), значение, proxy-group

### Лейаут страницы
- Реализовать **3 варианта** с переключателем:
  1. Вертикальный список — одна колонка, скролл, аккордеон
  2. Сетка карточек — grid 2-3 колонки, модальное/боковое раскрытие
  3. Как на Proxies — группы-карточки аналогично странице Proxies
- Пользователь выберет после визуальной оценки

### Drag-and-drop
- Перетаскивать **блоки между собой** и **отдельные правила внутри раскрытого блока**
- Драг за **всю карточку** целиком (без специальной иконки-ручки)
- Нумерация позиций (#1, #2...) — после перемещения номера пересчитываются
- **Полная свобода перемещения** — никаких жёстких ограничений
- **Предупреждения** при опасных операциях:
  - Перемещение MATCH с последней позиции — предупреждение с объяснением последствий
  - Перемещение «Персональных исключений» — предупреждение что они перестанут иметь наивысший приоритет
  - Предупреждения показываются **каждый раз** при такой операции

### Добавление правил
- Кнопка «Добавить правило» в раскрытом блоке → **диалог/модаль**
- Диалог с выбором типа правила (DOMAIN-SUFFIX, DOMAIN-KEYWORD, GEOSITE, RULE-SET, IP-CIDR...) и полем значения

### Смена proxy-group
- **Два уровня:**
  1. Dropdown на карточке — массовая смена для всех правил блока
  2. Dropdown на каждом правиле в раскрытом блоке — индивидуальная смена

### Создание нового блока
- Реализовать **2 варианта** с переключателем (пользователь выберет):
  1. Кнопка «Новый блок» → диалог с именем и выбором proxy-group
  2. Пустая карточка с inline-редактированием прямо на странице

### Удаление правил и блоков
- Диалог подтверждения **по умолчанию включён**
- Галочка «Больше не спрашивать» в диалоге — отключает подтверждения
- **Тумблер дублирован в настройках** — можно включить подтверждения обратно

### Сохранение
- Кнопки **Сохранить** и **Применить** как в Config Editor (фаза 5)
- **Бейдж-счётчик** на кнопке Apply — показывает кол-во несохранённых изменений («Применить (3)»)
- Изменённые блоки подсвечены визуально

### Diff-превью перед Apply
- **По умолчанию включено** — при Apply показывается diff изменений в секции rules
- Галочка «Больше не показывать» отключает превью
- **Тумблер дублирован в настройках** для повторного включения

### Отмена изменений
- Кнопка **Reset** — отменяет все несохранённые изменения, возвращает к последнему сохранённому состоянию
- **Ctrl+Z / Undo** — пошаговая отмена отдельных операций

</decisions>

<specifics>
## Specific Ideas

- Паттерн «галочка в диалоге + тумблер в настройках» используется для: подтверждения удаления, diff-превью. Единый подход ко всем опциональным подтверждениям
- Конфиг содержит ~30 proxy-group и ~300 правил — нужна виртуализация или пагинация для длинных списков
- В конфиге правила организованы комментариями-секциями (# --- YOUTUBE --- , # >>> ADULT и т.д.) — нужен парсер комментариев для режима «по секциям»

</specifics>
